## 这次我到底做了什么

这次我把作业做成了一个自己真的会用的东西：`OfferFlow`。它不是一个“演示用小壳子”，而是一个本地跑得起来的求职管理系统，用来记投递、跟进、面试复盘、公司信息和准备模板。

我一开始想得很现实：

- 投递不能散在 Excel、聊天记录、待办软件里
- 面试完不应该只剩一句“感觉还行”
- 公司信息、模板、历史记录最好能复用
- 最重要的是，这个东西要能在我自己的机器上稳定跑起来

我本来就知道怎么做 SDD，也知道 plan 模式适合把大问题拆开。只是这次作业只有两天，我没有刻意把它写成一套很“教科书”的流程，而是把这些方法放在真正有用的地方：先把方向收住，再把范围压小，最后一块一块落地。说白了，重点不是证明我会不会方法论，而是看看我能不能在时间很紧的时候，把方法论用得顺手。

当时我也很自然地开始想，接下来还能往哪些方向继续做。因为现在它只是一个雏形，还有很多地方值得优化，值得加功能，去更直接地解决用户痛点。比如我自己找工作的时候就很明显地遇到过这些问题：

1. 不知道西安不同行业里有哪些比较好的公司。我其实只知道一些知名公司，但一些不知名、但是很好的公司完全不知道，比如好的创业公司、小而美外企、行业里口碑不错但没那么大众的公司。
2. 即使知道某些公司，也要自己一个一个去查招聘官网。如果能直接点超链接跳到公司招聘网站或者公司官网，会方便很多。
3. 公司的内推码和内推信息很难找。如果能提供内推信息、内推联系人、内推方式、内推网站链接，或者其他一切内推相关信息就好了。
4. 某些公司或者某些岗位的面试经验很难找。如果能先看到可选公司列表，再直接看到不同职位的面经就很方便；或者更进一步，根据不同职位的 JD 自动生成面经，也会很有帮助。

所以我前期的目标并不是“把一个很大的系统做完”，而是先把这些真实痛点里最核心的一部分，做成一个能跑、能用、能继续迭代的工具。

## Day 1 — 2026-07-11

第一天主要就是定方向、搭骨架。

我先用 CC / Superpowers 做 brainstorming，把项目从一堆模糊想法里拉出来，最后决定还是做求职管理系统，而不是再做一个泛泛的 AI demo。方向确定后，我就开始写第一版规格，搭 Spring Boot + Thymeleaf + H2 的基础结构。

这一天最花时间的，其实不是功能本身，而是把前置问题先解决掉：

- Gradle 拉依赖慢，官方源经常超时
- 一开始总想先把页面做得好看一点，但很快发现底层模型和 CRUD 不稳，页面再漂亮也没用

所以我最后还是老老实实先把 Java 环境、Gradle 镜像、本地启动方式弄顺，再开始做实体、仓储、Service 和基础路由。

这一天的感觉很实在：**CC 不是只拿来“写代码”的，它更适合帮我把方向先收敛下来。**如果一开始就乱写，后面肯定会越做越散。

## Day 2 — 2026-07-12

第二天基本就是把项目真正做成型。

我把主路径补完整了：投递 CRUD、阶段流转、公司档案、面试复盘、仪表盘这些核心东西都做了。然后又顺手补了一轮比较实用的东西：

- UI 和交互打磨
- 404 / 500 异常页
- 逾期逻辑和日期校验
- 公共导航和移动端菜单
- 搜索、模板、seed、导出
- 页面冒烟测试和服务层测试

这一天下来，最真实的感受其实不是“功能好多”，而是“坑也不少”。有些问题单测是绿的，但我自己点页面还是会直接 Whitelabel。最开始我还以为是某个小地方没接好，后来才发现问题根本不在 Service，而是在模板表达式、变量命名这些更隐蔽的地方。

这里我最大的变化是：**我不再一上来就让 AI 乱修，而是先自己复现、看日志、分层，再让它帮我定位。**比如遇到 Whitelabel，我不是直接说“修一下”，而是先把它拆成几类可能：模板层、命名冲突、异常页缺失。这样 CC 才真的帮得上忙，不然它只会像个很快的执行器。

这一天我也明显感觉到，CC 的价值不只是“写得快”，而是**帮我更快形成判断**。我让它做的不是结论，而是把可能性、影响范围、修复顺序列清楚。这样我后面每次改动就能收得比较干净。

## 我在这两天里慢慢想明白的事

### 1. 原来还能这样

有几件事是我之前没那么认真想过的，这次做完之后确实有点“原来还能这样”的感觉：

- **先写 spec 再写代码**
  - 最早没使用SDD的时候很容易出现返工的情况
  - 这次先把需求写成 spec，再拆成 plan，开发反而更顺

- **手动 QA 比“测试过了就行”更有用**
  - 单测绿了，不代表主路径真没问题
  - 真正把 Whitelabel 这种问题暴露出来的，是我自己一条条点页面

- **JOURNAL 本身也能帮助打磨产品**
  - 写着写着我会发现，有些东西只是“代码做完了”，但还没变成真正有用的功能

### 2. 这玩意还不行

也有一件事让我觉得 CC 还没那么“神”：

- 当我只说“修一下 Whitelabel”时，它并不会自动知道问题到底在模板、Controller 还是异常处理上
- 如果不给足上下文，它就容易往局部改动上走，而不是主动把根因翻出来

所以我现在的感觉是：**CC 很强，但它不是替我做定位的。**复杂问题还是得先由人把范围收窄，再让它去提速。

另外，Superpowers 的流程我也有一点很实际的感受：对大改动它很有帮助，但对特别小的修补来说，完整走一遍 brainstorming 和 plan 会显得有点重。可我也知道这套流程是干什么的，所以我没有因为“嫌麻烦”就把它当成摆设，而是把它用在真的值得做设计的地方。小修补就尽量轻一点，大改动还是老老实实走完整流程。

## 如果再给我一周，我会怎么改

如果这不是两天收尾，而是真的再给我一周，我大概率会继续做这几件事：

1. 加登录和多用户隔离，让它不只是单机工具
2. 把数据层从 H2 换成 MySQL，做得更稳一点
3. 加更明确的提醒机制，比如待跟进提醒、面试后 24 小时复盘提醒
4. 做一个更适合手机的小输入页，降低“我懒得记”的阻力
5. 继续把 JOURNAL 和 demo 讲述打磨得更自然一点

## 换个角色思考

如果我是一个第一次接触 CC 的同事，我会给他配的东西其实不会很多。

我会先给他一个固定 skill，让它负责把日常工作变成结构化输入，比如“记录事项 → 生成下一步动作”。再给一个 sessionStart hook，每次打开就提醒当天最重要的事情，防止工具最后变成“想起来才用一下”的聊天窗口。最后如果需要，再加一个轻量 workflow，把“想法 → 拆解 → 执行 → 回顾”这条链路接起来。

原因其实很简单：新手最大的难点不是不会提问，而是不会把自己的工作拆成 AI 能协作的形状。skill 和 hook 的价值就在这里，它们不是为了炫技，而是把工作方式固定下来。

## 写在最后

我认为AI更像一个能帮我组织工作方式的协作对象。它最有价值的地方，不是替我思考，而是把我已经想清楚的东西放大、整理、提速。

所以最后我愿意花时间去打磨 `README.md`、`JOURNAL.md` 和 `.claude`，因为这几个东西本身就是这次作业最重要的证据链：它们不只是说明“我做了什么”，也在说明“我是怎么做的”。

因个人求职进度安排，我需在周一前确认相关 offer 意向。为能同步评估本次机会、不耽误双方节奏，我主动加快了项目进度，在完整覆盖所有考核要求的前提下，于 2 天内完成了全部开发与文档工作，提前提交作业。

如果真的再给我一周我大概率会把项目做的更大一点，例如做前后端分离、做成微服务架构、添加账户管理体系、添加简历编辑器、通知与提醒渠道、集成微信机器人等等。 我的设想是最终可以把`Offerflow`这个工具衍生成为一个持续项目，把所有动作串起来：投递、跟进、面试、复盘、Offer、入职，让用户一眼就能看到整个过程。
